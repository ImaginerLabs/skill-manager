## 成功与亮点

### 1. 类型系统扩展的向后兼容设计（AD-43）

`WorkflowStep` 类型扩展采用了精心设计的向后兼容策略：
- `type` 字段新增，但旧格式文件解析时自动推断（`step.skillId ? 'skill' : 'custom'`）
- `skillId` 和 `skillName` 从 `string` 改为 `string | null`，使用 `nullable()` 而非 `optional()`，语义更明确
- `loadWorkflow` 自动为缺少 `type` 字段的步骤补全默认值

这确保了新旧工作流文件无缝共存。

### 2. CustomStepCard 视觉区分清晰

自定义步骤卡片与 Skill 步骤卡片的视觉区分做得很好：
- **边框**：虚线 `2px dashed #475569`（vs Skill 步骤的实线）
- **图标**：✏️ 铅笔（vs Skill 步骤的 ⚡ 闪电）
- **背景**：微弱斜线纹理
- **Textarea**：自动扩展（1~6 行），`font-mono`，无边框融入卡片

这种视觉区分让用户一眼就能区分两种步骤类型。

### 3. "Pushy" Description 触发策略

`generatePushyDescription` 函数实现了 skill-creator 规范推荐的"pushy"触发策略 — 不仅描述功能，还主动包含触发场景和关键词。这提升了工作流在 Claude 生态中的触发表现。

### 4. Frontmatter 完整对齐 skill-creator 规范（AD-44）

生成的工作流文件 Frontmatter 包含了 skill-creator 规范要求的所有字段：
- `name`：工作流名称
- `description`：pushy description（用户输入优先，否则自动生成）
- `category: workflows`：固定分类
- `type: workflow`：固定类型
- `tags`：从步骤 Skill 名称聚合 + "workflow" 标签

### 5. 拖拽排序混合支持

自定义步骤与 Skill 步骤可以混合拖拽排序（复用 `@dnd-kit/sortable`），sortableIds 使用 `step.type + '-' + step.order` 作为唯一 ID，避免了 `skillId` 为 `null` 时的 ID 冲突。

### 6. WorkflowPreview 验证逻辑适配

`getDisabledReason` 函数适配了自定义步骤：
- 允许纯自定义步骤工作流（无 Skill 步骤也可生成）
- 新增验证：自定义步骤描述不能为空

## 挑战与改进空间

### 1. Story 文件状态更新遗漏

V2 Epic 4 的 3 个 Story 文件在实现完成后没有更新状态（停留在 `ready-for-dev`），Tasks 也没有勾选，Dev Agent Record 为空。这与 sprint-status.yaml 中的 `done` 状态不一致。

**教训：** 这是 Epic 8 回顾中提出的"Epic 收尾三步检查"行动项的再次违反。需要更严格地执行 Story 文件状态同步。

### 2. 自定义步骤描述无字数限制

CustomStepCard 的 Textarea 最大 6 行，但没有字符数限制。如果用户输入极长的描述，生成的工作流文件可能不够简洁。可以考虑添加字符数限制或提示。

### 3. generatePushyDescription 的中文硬编码

`generatePushyDescription` 函数中的模板文字是中文硬编码（"组合...的...工作流"、"当用户需要...时使用此工作流"），没有走 i18n。虽然生成的工作流文件主要面向中文用户，但如果未来需要支持英文工作流生成，需要国际化。
