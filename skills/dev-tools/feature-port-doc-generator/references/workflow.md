# 详细工作流程说明

## 完整决策树

```
用户触发 Skill
│
├─ 提供了 commit ID？
│  ├─ Yes → 调用 MCP 获取 commit 信息 + diff
│  └─ No  → 用户提供了文件路径？
│           ├─ Yes → 读取文件内容 → 查找相关 commit
│           └─ No  → 向用户询问：commit ID 或文件路径
│
↓
分析需求改造意图（必须完成，不可跳过）
│
├─ 意图清晰？
│  ├─ Yes → 评估移植复杂度
│  └─ No  → 向用户确认需求背景
│
↓
评估移植复杂度（简单 / 中等 / 复杂）
│
↓
生成移植文档（按模板填写）
│
↓
输出文档 → 询问用户是否调整 → 保存 MD 文件
```

---

## 第一阶段详解：获取变更信息

### 情况 A：使用 MCP 工具读取 commit

```
# 步骤 1：获取 commit 基本信息
use_mcp_tool(
  serverName: "gongfeng_sys",
  toolName: "get_commit_info",
  arguments: {
    project_id: "<项目ID或路径>",
    commit_sha: "<commit ID>"
  }
)

# 步骤 2：获取 diff
use_mcp_tool(
  serverName: "gongfeng_sys",
  toolName: "get_commit_diff",
  arguments: {
    project_id: "<项目ID或路径>",
    sha: "<commit ID>"
  }
)
```

**注意**：如果用户没有提供 project_id，需要先询问，或根据当前工作目录的 git remote 推断。

### 情况 B：从文件路径查找 commit

```bash
# 查找指定文件的最近提交记录
git log --oneline -10 -- <文件路径>

# 查看某次提交的详细 diff
git show <commit_id>

# 查看文件的完整变更历史
git log -p -- <文件路径>
```

**操作步骤**：

1. 执行 `git rev-parse --show-toplevel` 获取项目根目录的绝对路径
2. 读取用户指定的文件内容，了解当前状态
3. 执行 `git log` 获取该文件的提交历史
4. 根据 commit message 判断哪次提交与需求相关
5. 获取该 commit 的 diff，进入第二阶段分析

### 情况 C：无法获取 commit，直接分析文件

当无法获取 commit 历史时（如文件是新建的、或 git 历史不可用）：

1. 读取文件完整内容
2. 根据代码注释、函数命名、业务逻辑推断改造意图
3. 在文档中注明"基于文件内容分析，无 commit 历史参考"

## 第二阶段详解：分析需求改造意图

### 需求类型判断

| 类型         | 判断依据                             | 文档重点                     |
| ------------ | ------------------------------------ | ---------------------------- |
| **功能新增** | 新增文件、新增函数、新增页面         | 功能描述、接口依赖、UI 变化  |
| **Bug 修复** | commit message 含 fix/修复，改动较小 | 问题根因、修复逻辑、影响范围 |
| **性能优化** | 涉及缓存、懒加载、防抖节流等         | 优化前后对比、性能指标       |
| **重构**     | 大量文件移动、函数重命名、结构调整   | 重构目标、文件映射关系       |
| **样式调整** | 仅涉及 CSS/样式文件                  | 视觉变化、设计规范           |
| **配置变更** | 涉及配置文件、环境变量               | 配置项说明、默认值           |

### 阅读 diff 的方法

拿到 diff 后，按以下顺序分析：

1. **先看 commit message**：通常直接说明了改造目标
2. **统计改动文件数量**：判断需求规模（单文件 vs 多文件）
3. **识别改动类型**：
   - 新增文件（`new file mode`）
   - 删除文件（`deleted file mode`）
   - 修改文件（`@@` 标记的变更块）
4. **逐文件分析**：
   - 删除了什么（`-` 行）
   - 新增了什么（`+` 行）
   - 改动的上下文（周围未变化的代码）

### 依赖识别清单

分析 diff 时，重点关注以下依赖引入：

```
□ import 语句：是否引入了新的 npm 包？
□ 工具函数：是否调用了新的工具函数（utils/helpers）？
□ 组件引用：是否使用了新的 UI 组件？
□ API 接口：是否新增了后端接口调用？
□ 类型定义：是否新增了 TypeScript 类型？
□ 全局状态：是否使用了新的 Redux/Pinia store？
□ 环境变量：是否依赖新的环境变量配置？
□ 样式变量：是否使用了新的 CSS 变量或主题 token？
```

## 第三阶段详解：评估移植复杂度

### 复杂度评分标准

| 维度       | 简单（1分） | 中等（2分） | 复杂（3分） |
| ---------- | ----------- | ----------- | ----------- |
| 改动文件数 | 1-2 个      | 3-5 个      | 6个以上     |
| 新增依赖   | 无          | 1-2 个      | 3个以上     |
| 逻辑关联性 | 独立        | 有关联      | 强耦合      |
| 架构影响   | 无          | 局部        | 全局        |
| 前置条件   | 无          | 少量        | 较多        |

**总分判断**：

- 5-7 分：简单，可直接参考代码实现
- 8-11 分：中等，按文件顺序逐步移植
- 12-15 分：复杂，需分阶段移植，先完成前置条件

### 前置条件识别

移植前，接收方项目需要具备：

```
□ 技术栈兼容性（React/Vue 版本、TypeScript 版本等）
□ 依赖包已安装（列出具体包名和版本）
□ 基础组件/工具函数已存在（列出具体名称）
□ 后端接口已就绪（列出接口路径和参数）
□ 配置项已添加（列出配置项名称和说明）
```

## 第四阶段详解：生成移植文档

### 文档生成原则

1. **路径必须绝对**：文档中所有文件路径必须使用绝对路径，禁止使用 `src/xxx` 这类相对路径
2. **接收方视角**：假设读者不了解原项目，所有术语需要解释
3. **步骤可执行**：每个步骤必须具体到“打开哪个文件（绝对路径）、修改哪里、改成什么”
4. **代码示例完整**：关键代码片段必须包含足够的上下文，不能只贴片段
5. **风险提前说明**：常见问题和注意事项放在步骤前，而非步骤后

### 文件顺序建议

多文件改动时，建议按以下顺序在文档中描述：

1. **类型定义文件**（`.d.ts`、`types.ts`）— 其他文件依赖它
2. **工具函数/Hook 文件** — 业务逻辑依赖它
3. **状态管理文件**（store、reducer）— 组件依赖它
4. **核心业务逻辑文件** — 主要改动所在
5. **UI 组件文件** — 展示层
6. **配置文件** — 环境相关
7. **测试文件** — 可选，最后处理

### 代码示例写法规范

````markdown
**改动前**：
\```typescript
// 原始代码（保留足够上下文，至少3行）
const result = someFunction(param);
\```

**改动后**：
\```typescript
// 修改后的代码（说明改动原因）
// 增加空值判断，防止 param 为 null 时报错
const result = param ? someFunction(param) : defaultValue;
\```
````

## 第五阶段详解：输出与保存

### 文档保存路径建议

```bash
# 使用绝对路径保存，默认保存在当前目录
# 先获取当前目录的绝对路径
pwd
# 示例输出：/Users/alex/Desktop/Github/my-project

# 最终保存路径（绝对路径）
/Users/alex/Desktop/Github/my-project/port-doc-<需求简称>-<日期>.md

# 示例
/Users/alex/Desktop/Github/my-project/port-doc-cart-quantity-fix-20260330.md
```

### 输出后的确认流程

```
1. 输出完整文档内容
2. 询问："文档已生成，是否需要调整某个章节？"
3. 根据用户反馈修改
4. 询问："是否保存为 MD 文件？保存路径是？"
5. 保存文件
```

### 常见用户反馈处理

| 用户反馈         | 处理方式                       |
| ---------------- | ------------------------------ |
| "步骤太简略"     | 展开具体步骤，增加代码示例     |
| "依赖写漏了"     | 补充到依赖清单章节             |
| "背景描述不准确" | 重新确认需求背景，修改概述     |
| "需要加截图说明" | 在对应步骤后添加截图占位符说明 |
| "需要中英文对照" | 在关键术语后添加英文原文       |
